§ 01 · QuickstartYour first request, in ninety seconds.

The Stratum API is a thin, opinionated layer over the same tools the Console runs against your tenant. If you can ask a question in the Console, you can ask it programmatically. The verbs are the same; only the response is colder.

Every request is authenticated with a scoped JWT issued by your tenant's OAuth client. The example below assumes you have already exchanged a client_credentials grant for an access token and exported it as $STRATUM_API_KEY. If you don't have a client yet, write to api@stratum.ai — we'll provision one for your design-partner sandbox the same business day.

The canonical first call is a cohort query against the people graph. Pick a slice you care about (here: senior engineering in EMEA), ask for it, and read the response. Every read endpoint returns a row payload and a citation_pack — the same provenance bundle the Console uses to defend its answers in the boardroom.

Below: a curl request, the JSON response you should expect, and a brief annotation of what each top-level field contains. Reading these three blocks is enough to start integrating. The remainder of this document is reference material you'll consult once, in anger, when something at 2am has stopped behaving the way you remember.

curl · request
curl https://api.stratum.ai/v1/people/query \
  -H "Authorization: Bearer $STRATUM_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Stratum-Version: T+7d" \
  -d '{
    "filters": { "region": "EMEA", "level_in": ["IC5", "IC6", "M3"] },
    "limit": 20,
    "sort_by": "comp_ratio",
    "sort_desc": true
  }'
JSON · response (truncated)
{

  "truncated": true,
  "rows": [
    {
      "id": "emp_8a2c14e0",
      "display_name": "A. Achterberg",
      "title": "Staff Engineer, Platform",
      "level": "IC6",
      "department": "Engineering",
      "team": "Platform · Identity",
      "location": "Amsterdam",
      "country": "NL",
      "region": "EMEA",
      "tenure_years": 5.4,
      "comp_total": 218000,

      "comp_ratio": 1.07,

What this query returns

The cohort is the set of EMEA-based individual contributors and first-level managers at career bands IC5 through M3 — 387 people on the date this query ran. rows is a representative sample of up to 20, sorted by comp_ratio descending; if the cohort exceeds limit, the response includes one or two edge-case rows so you don't blind yourself to the long tail. Use cursor for stable pagination.

The field schema is identical to the Console's internal projectRow view: stable ids, display names (no PII like email), levelled compensation against the band's 50th percentile, a tenure value, a performance score on the 1-to-5 scale, and a flight-risk score paired with its banded label. Gender is not returned on individual rows; you'll only see it surface in cohort-level aggregates with cell-size suppression, never in row payloads. This is a deliberate, non-configurable constraint.

The citation pack is the per-response provenance bundle. It carries the method, the controlled dimensions, the cohort size, a confidence label, and the snapshot identifier of the source data. Citation packs are immutable and audit-logged for eighteen months. You can replay any one of them via GET /v1/citations/{id} — useful for compliance, useful for reproducing the exact answer your CHRO read in a board deck six weeks ago.

§ 02 · AuthenticationOAuth 2.0 with scoped JWTs.

Stratum runs the same auth model whether you're calling from a server, an interactive web app, or an internal AI agent. Three flows; one token type; scopes that map exactly to your tenant's RBAC.

Flows

client_credentials. Server-to-server. The most common flow. You exchange a client id and secret for a short-lived access token at https://auth.stratum.ai/oauth/token. Tokens are JWTs signed with RS256, expire in 3600 seconds, and carry the scopes you requested. Refresh by re-exchanging — there are no refresh tokens for machine credentials by design. Rotate the client secret every 90 days, or every 30 days for clients that hold admin.tenant.

authorization_code with PKCE. Interactive applications (custom dashboards, internal tools) embed the standard OAuth handshake. Stratum hosts the consent screen at https://auth.stratum.ai/authorize and returns to your registered redirect URI with an authorization code that exchanges for an access token (1 hour) and a refresh token (90 days). Per-user scopes gate exactly the slice of the people graph the signed-in user could see in the Console — your code does not need to re-implement permissions.

BYOA — bring-your-own-agent. The flow for embedding Stratum behind a customer-owned LLM agent. You issue a scoped JWT to the agent at runtime (with a 15-minute TTL, recommended) and rotate per session. The agent presents the Stratum JWT to our API; it presents your Anthropic/OpenAI/etc. credentials to its own model provider. The two never meet on our side. The Console itself uses exactly this pattern; the section below explains why that matters.

curl · client_credentials
curl https://auth.stratum.ai/oauth/token \
  -u "$STRATUM_CLIENT_ID:$STRATUM_CLIENT_SECRET" \
  -d "grant_type=client_credentials" \
  -d "scope=people.read aggregate.read citations.read"
JSON · token response
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InN0cmF0dW0tMjAyNi0wNCJ9.eyJpc3MiOiJodHRwczovL2F1dGguc3RyYXR1bS5haS8iLCJzdWIiOiJjbGllbnRfMDFIWko...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "people.read aggregate.read citations.read",
  "tenant_id": "tnt_lattice_corp"
}

Scopes

ScopeGrants
people.readRead rows from the people graph; cohort queries, single-entity lookups, edge traversal.
aggregate.readRun grouped statistics, distributions, and pay-equity analyses. Cell-size suppression is always applied.
decisions.writeCreate decisions that write back to a connected system. Requires an attached citation_pack and an approver_id. Gated to Platform + Decisions plan.
citations.readRetrieve, list, and replay citation packs. Needed for audit and reproducibility tooling.
webhooks.manageSubscribe to and manage webhook endpoints for tenant events.
admin.tenantAdminister tenant configuration, OAuth clients, scope mappings, retention policies. Sensitive; 30-day secret rotation.

Token rotation

Access tokens (JWTs) expire after one hour and you should treat them as disposable. Client secrets rotate on a default 90-day cadence, surfaced as a calendar warning in the developer dashboard 14 days before expiry. Any client that holds decisions.write or admin.tenant is required to rotate every 30 days; a missed rotation auto-suspends the client and emits a webhook (client.secret.expired) so your on-call sees it before your customers do.

A note on the BYOK pattern used by the Console When you embed Stratum in a browser-facing surface (like our own Console), you may use a customer-issued Anthropic API key alongside a Stratum scoped JWT. The Stratum JWT controls access to your people graph; the Anthropic key handles the reasoning model. The two credentials are presented to different endpoints, on different origins, with different lifetimes — and we never see either. The Console runs entirely in your browser; both keys live in your localStorage, neither traverses our servers. This is the same posture you should adopt for any agent you build on the Stratum API.

§ 03 · ConceptsFour ideas that recur in every endpoint.

Read this section once. After that, the endpoint reference will feel like a checklist instead of a manual.

The people graph

Stratum models your workforce as a typed graph, not a flat employee table. Entities are the nouns: Employee, Position, Team, Location, CompensationBand, PerformanceReview, SkillTag. Edges are the verbs that connect them: reports_to, manages, member_of, compensated_at, performed, located_at, previously_held. Every edge carries a validity interval and a provenance record — the system that asserted it, the timestamp it was asserted, the confidence score at assertion time.

Entity resolution is the boring miracle that makes the graph trustworthy. When Workday calls someone "Anna J. Achterberg", Greenhouse calls her "AJ Achterberg", and Slack calls her "anna.achterberg@…", Stratum resolves them to a single entity at a tunable similarity threshold (default 0.94). Below threshold we hold the candidates open as entity_conflict_unresolved and route to a human reviewer rather than silently merging. Identity provenance — which fields came from which connected system, and when — is exposed on every row under _provenance when you set include=provenance on a query.

Decisions and citations

Every read returns a citation_pack. It carries method (a human-readable description of what was computed), cohort_size (n of the population used), dimensions_controlled (the variables held constant, where applicable), and a confidence label of high, moderate, or low. Suppressed cells, outlier handling, and any methodology notes flow into caveats.

A decision is a write-back to a connected system: a comp adjustment posted to Workday, a role change pushed to your HRIS, a promotion case opened in your performance tool. Decisions are created via POST /v1/decisions and require both an approver_id (the human accountable for the decision) and a citation_pack id (the evidence the decision was based on). Citation packs are immutable; you cannot edit one after creation. They are retained, fully indexed and replay-able, for eighteen months — the EU AI Act audit window, with a buffer.

Filter grammar

Every read endpoint accepts the same filter shape. The grammar is intentionally narrow — we'd rather support every important question well than every possible query badly. The canonical fields and operators:

JSON · filter shape
{
  "department": "Engineering",            // eq, scalar
  "department_in": ["Engineering", "Data"], // in
  "region": "EMEA",                     // one of NA, EMEA, APAC, LATAM
  "region_in": ["EMEA", "APAC"],
  "country": "NL",
  "location": "Amsterdam",
  "team": "Platform · Identity",
  "level": "IC6",
  "level_in": ["IC5", "IC6", "M3"],
  "employment_type": "FT",
  "flight_risk_band": "high",          // low | moderate | high
  "flight_risk_band_in": ["moderate", "high"],
  "is_manager": true,
  "promotion_eligible": true,
  "comp_ratio_lt": 0.92,             // lt / gt / lte / gte
  "comp_ratio_gt": 1.05,
  "comp_total_lt": 240000,
  "comp_total_gt": 120000,
  "performance_gte": 3.5,
  "performance_lte": 4.8,
  "tenure_gt": 2,                     // years; tenure_lt also supported
  "span_gte": 5,
  "span_lte": 12
}

Operators supported: eq (implicit, scalar), in (suffix _in on array fields), lt / gt / lte / gte (suffix on numeric fields), between (use _gt + _lt), like (string prefix; on title and team only), is_null (suffix _is_null: true), and not (wrap any filter object in {"not": { … }}). Multiple keys at the same level AND together. For OR semantics, use the cohort-join shape below.

JSON · cohort join (OR)
{
  "any_of": [
    { "region": "EMEA", "flight_risk_band": "high" },
    { "region": "APAC", "performance_gte": 4.5 }
  ]
}

Time fidelity

Stratum is event-sourced. Every change to the graph — a new hire, a comp revision, a manager swap, a team rename — is recorded as an immutable event with a monotonic timestamp. The current state of the graph is just the fold of every event up to now; the historical state is the fold up to whatever instant you ask for.

Every read endpoint accepts as_of, an ISO 8601 timestamp. Pass it and the graph is materialised at that moment; compensation, headcount, manager relationships, even entity-resolution decisions are evaluated as they stood. This is not a snapshot pulled from a backup — it is a first-class query. The same endpoint that answers "What is our pay gap in EMEA today?" answers "What was it on March 31, 2025, before the EU pay-transparency deadline?" with the same SLA.

JSON · time-travel query
{
  "as_of": "2025-03-31T23:59:59Z",
  "filters": { "region": "EMEA" },
  "dimensions": ["level", "region"],
  "minimum_n_per_cell": 10
}

§ 04 · EndpointsSix verbs. Same surface as the Console.

Every endpoint takes JSON, returns JSON, attaches a citation pack, and respects scope. The cards below are intentionally repetitive — once you know one, you know all of them.

Filter and return rows from the people graph. Up to 200 rows per request with cursor pagination; representative-sampling kicks in when limit is below the matched count and preserves edge cases on the tail.

Parameters

limitintegeroptional · default 20 · max 200Maximum rows in rows. Cohort total is always returned in total_matched.
sort_bystringoptionalField to sort by. e.g. comp_ratio, flight_risk, tenure_years, performance_score.
sort_descbooleanoptional · default falseDescending sort.
cursorstringoptionalOpaque cursor returned in a prior response; preserves the snapshot used for the original query.
as_ofstring · ISO 8601optionalMaterialise the graph at this instant. Defaults to "now" (latest snapshot).
includearray<string>optionalAdditional fields: provenance, edges, audit.

payloadobjectrequiredType-specific body. See per-type schemas in the developer dashboard.
approver_idstringrequiredThe human accountable for the decision (an emp_* id with the appropriate role).
citation_pack_idstringrequiredReference to the citation_pack that supports the decision. Must be ≤ 30 days old.
target_systemstringrequiredDestination connector: workday, bamboohr, ukg, greenhouse, etc.
idempotency_keystringoptional · recommendedClient-generated key; same key + same body = same decision, never duplicated.

Decisions are irrevocable from Stratum's side Once a decision is queued for write-back, you cannot delete it through this API. You can issue a compensating decision (e.g. revert a comp_adjustment) and Stratum will record both, in order, in the audit log. The connected target system's policies still apply on write — Workday may reject a comp change that violates a band, and the decision will surface in status writeback_rejected with the upstream error in writeback_error.

Error semantics

400 invalid_filter (malformed payload); 403 scope_insufficient if the caller lacks decisions.write or if the approver_id isn't authorised for this decision type; 409 entity_conflict_unresolved if subject_id is in entity-resolution review; 422 if the citation_pack is older than 30 days or doesn't cover the subject.

Rate limit cost

5 units per request. Decisions count against the write-rate limit (50 req/min default).

§ 05 · WebhooksPush notifications for people-graph events.

Stratum sends signed webhooks when entities in the graph change or when long-running evaluations finish. Subscribe per event type, per environment, per URL.

Event types (v0.4)

EventFires when
employee.createdA new employee entity is resolved (typically from an HRIS create).
employee.updatedAny field on an employee changes (level, location, comp_total, etc.).
employee.terminatedAn employee's terminated_at is recorded.
manager.changedA reports_to edge moves between entities.
comp.review.completedA compensation review cycle finishes for a cohort.
decision.createdAny decision is created via POST /v1/decisions.
decision.writeback_failedA queued decision could not be written to its target system.
eval_set.evaluation_completedA scheduled eval-set run completes. Payload includes pass-rate and drift metrics.
client.secret.expiredAn OAuth client secret has expired and the client is now suspended.

Delivery is at-least-once with exponential back-off and a maximum of 8 attempts over 24 hours. Acknowledge with any 2xx; we treat 410 as a permanent unsubscribe. Use Stratum-Delivery-Id to deduplicate on your side. Verify Stratum-Signature by computing HMAC-SHA256 over {timestamp}.{body} with your endpoint's signing secret and comparing in constant time.

§ 06 · ErrorsStandard HTTP, plus a handful of Stratum-specific codes.

Every error response is JSON, with a stable error machine-readable code, a human-readable message, a request_id for support, and (where useful) a retry_after_ms hint.

JSON · error shape
{
  "error": "cohort_too_small",
  "message": "Matched cohort (n=3) is below the minimum cell size for this operation (5).",
  "request_id": "req_01HZK92B7QABYMFC8NTKDXWRZP",
  "hint": "Widen the cohort by relaxing one filter, or lower min_cell_n if your tenant permits."
}
CodeNameWhen it happensWhat to do
400invalid_filterFilter object contains an unknown field, an unsupported operator, or a value of the wrong type.Validate against the filter grammar (§ 03). The message names the offending key.
401missing_or_invalid_tokenAuthorization header is missing, malformed, expired, or signed by a key Stratum doesn't recognise.Re-exchange your client credentials at /oauth/token; verify clock skew < 60s.
403scope_insufficientToken is valid but doesn't carry the scope this endpoint needs (e.g. decisions.write on a read-only client).Re-issue the token with the required scope. Don't widen scopes preemptively.
404entity_not_foundThe path resource (citation, decision, employee) does not exist or is not visible to this tenant.Confirm the id. Cross-tenant access is reported as 404, not 403, by design.
409entity_conflict_unresolvedEntity resolution flagged the subject for human review; writes against it are blocked.Resolve the conflict in the admin console, or call back after the entity.resolved webhook.
422cohort_too_smallThe post-filter cohort is below the minimum cell size for the requested operation.Widen the cohort. For aggregates, lower min_cell_n if tenant policy permits.
429rate_limitedRate budget exceeded for this client. Retry after the duration in retry_after_ms.Back off and retry. Use the X-RateLimit-* headers to pace yourself.
500internalUnexpected server error. Stratum has been alerted.Retry idempotent requests with exponential back-off. For writes, check the audit log before retrying.
503upstream_unavailableA connected source-of-truth system (Workday, BambooHR, etc.) is unreachable; read continues from the last snapshot.Reads still succeed against cached snapshots. Retry writes when connector.recovered webhook fires.
524graph_traversal_timeoutA graph traversal exceeded the per-request budget (default 8s).Narrow the cohort, or contact us to raise the budget for a specific use case.

§ 07 · Rate limitsTwo budgets, generous bursts, per-tenant overrides.

Reads run on one budget; writes on another. Burst capacity is 4× the per-minute rate. Enterprise contracts can negotiate higher steady-state limits per endpoint family.

ClassSteady-stateBurstEndpoints
Read200 req/min800 req/min for 15speople/query, aggregate, distribution, analytics/pay_equity, citations/*
Write50 req/min200 req/min for 15sdecisions
Auth30 req/min90 req/min for 15soauth/token, webhooks management

Every response carries X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset (a Unix timestamp in seconds). When you hit the limit you'll receive a 429 rate_limited with a retry_after_ms value in the body and the Retry-After header in seconds.

Endpoints have unit costs (see each endpoint's "Rate limit cost" note). Costs roll into the same budget — an aggregate request costs 2 units against the read budget, a decision costs 5 against the write budget. The Console runs against the same budgets your engineering team does; if the Console is sluggish, check whether a script is in a loop.
