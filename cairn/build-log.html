Migration script — the relevant edge case

For the record, because the schema review surfaced it and I want it in the log rather than buried in a PR description: the v5 migration treats every existing artifact node as potentially evidence. A natural read is that we should backfill cairn_stone nodes from historical artifacts at migration time. We will not. Backfill is a future-Forge problem; the v5 migration only adds the schema. Stones get placed forward from W03 when Talus starts running. Reasoning: backfilled stones lack consent-at-placement and lack a contemporaneous attestor, which violates 014-T's observable axis. A stone we cannot defend the provenance of is worse than no stone.

# /console/data/_migrations/v5_cairn.py — draft, W01 D1 # additive only. no UPDATE on existing rows. no DROP. no ALTER on existing columns. def migrate(graph): graph.add_node_type("cairn_stone", schema=cairn_stone_schema) graph.add_node_type("skill_gap", schema=skill_gap_schema) graph.add_node_type("trajectory", schema=trajectory_schema) graph.add_node_type("attestation", schema=attestation_schema) graph.add_edge_type("closes", src="trajectory", dst="skill_gap") # no backfill. stones place forward from W03 when Talus runs. # reasoning: backfilled stones lack consent-at-placement → violates 014-T observability. return graph.version_bump("v4", "v5")

Risks tracked today

  • Risk A · High · Counsel-blocking AI Act Annex III classification. Architecture §V states our position that Cairn is a recommendation system, not an assessment system. Counsel needs to ratify that posture against the August 2026 Annex III deadline before we sign any EU design partner. The Cordova conversation may force the question earlier than W11 — Cordova is US-headquartered with EU operations. Mitigation: I am writing the procurement-grade compliance memo this week (target W01 D4) so Counsel has the working document to react to. If Counsel disagrees with the recommendation-system framing, we have to rebuild the Hearth verification surface before W07 — material schedule risk.
  • Risk B · High · Eval-blocking Activation-tier perverse-incentive. The hybrid pricing model bills on activated trajectories. Talus is the agent that opens them. A naive Talus will over-detect gaps to generate billable trajectories. Kernel's gap-fabrication probe class (28 probes, bar ≤ 0.02 false-positive rate) is the bounding mechanism — but the threshold needs empirical validation against real customer data before we sign Cordova. The synthetic corpus may be too clean. I want a calibration run on Cordova's first month of read-only signals before any trajectory opens in their tenant. Logging this so we do not forget.
  • Risk C · Medium · Yield-dependent Hearth verification needs work-artifact instrumentation we do not have on every connector. Hearth's stage-4 verification reads from the same work-signal stream Talus uses for stage-1 detection. We have artifact-grade instrumentation for the Recruiter / Tenure connector set (GitHub, Linear, Slack, GDrive, Confluence, Notion). We do not have it for Salesforce, ServiceNow, Workday, or the Microsoft 365 stack — and Cordova runs heavily on Microsoft 365. Flagged for Yield as a W04 dependency. If Hearth cannot read work-artifacts on Cordova's surfaces, verification reduces to manager-signoff, which works but is weaker.
  • Risk D · Watch · Schema Pillar's attestation-signature-as-blob objection. Deferred to W02. Not blocking. Logged because schema decisions are sticky — once stones are placed against the v5 layout, restructuring attestation.signature is a real migration. We have two weeks. I would rather get this right than ship.
