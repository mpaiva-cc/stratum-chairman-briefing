§ 01The schema, in plain language

What each node and edge means.

The graph is small, and so is the vocabulary. Four node types. Four edge types. The combinatorics live in the data, not the schema.

Employee node node · person

A person on the payroll. Has a manager, a level, a comp band, and a tenure. Required to be connected to exactly one manager (except the CEO/CHRO). May be a member of one or more teams.

Contractor node node · person

A person with limited tenure, no comp band, and a reference to a payroll vendor. Has an engagement window (valid_from, valid_until). Reports to an employee — never to another contractor. Cannot manage anyone.

Agent node node · system

An AI agent: ours or a customer's. Has a creator (the human who instantiated it), an oversight policy (who can halt it, when), and an access scope (which slices of the graph it can read or modify). First-class. Audited like a person.

Team node node · grouping

A logical grouping — engineering, sales, customer success, ops. Many-to-many with employees, which is the whole point: matrix organizations are supported because a person can be a member of multiple teams without any of them being a lie.

reports_to edge edge · 1 → 1

Directed: an employee → their manager. Required for every employee on the payroll. Versioned — when someone changes managers, the old edge is closed (valid_until) and a new one opens. The historical chain is preserved.

manages edge edge · 1 → N

The inverse of reports_to, materialized for query convenience. Never written directly — derived. Surfacing it as a first-class edge means "span of control" is one hop, not a self-join.

member_of_team edge edge · N → N

An employee → a team. Many such edges per employee. Carries attributes: role (lead, member, advisor), allocation (1.0, 0.5, 0.2), since. The matrix lives here.

compensated_at_band edge edge · 1 → 1 (current)

An employee → their comp band, with effective dates. Closed and re-opened on promotion or market adjustment. Pay equity queries traverse this edge — they never read a column.

§ 02The schema, in formal notation

For the engineers.

The same idea, with types. JSON-Schema-ish, abbreviated. The full schema is published at /api/schema.v3.json and ratified fortnightly in the Compact. 

// ── Node types ────────────────────────────────────────────────────────

Employee {
  id:               string,                // stable, opaque, never reused
  display_name:     string,
  level:            "IC1" | "IC2" | ... | "IC7" | "M1" | ... | "M5",
  manager_id:       Employee.id | null,   // null only at the top of the tree
  team_ids:         [Team.id],            // 1..N · matrix supported
  comp_band_id:     CompBand.id,
  hire_date:        date,
  terminated_at:    date | null,            // null = active
  comp_total:       number,                // in band currency · annualized
  region:           "NA" | "EMEA" | "APAC" | "LATAM",
  flight_risk:      "low" | "moderate" | "high",
}

Contractor {
  id:               string,
  display_name:     string,
  vendor_ref:       string,                // e.g. "magnit:CC-4471"
  engagement_start: date,
  engagement_end:   date | null,            // auto-expires
  sponsor_id:       Employee.id,           // who owns the engagement
  rate:             number,                // hourly, in vendor currency
  comp_band_id:     null,                  // contractors are NOT banded
}

Agent {
  id:               string,
  display_name:     string,
  kind:             "stratum-builtin" | "customer-defined",
  creator_id:       Employee.id,           // always a human
  oversight_policy: OversightPolicy.id,    // halt-conditions live here
  access_scope:     [Scope],               // e.g. "read:people.region=EMEA"
  last_review:      date,                  // 90-day cadence enforced
}

Team {
  id:               string,
  display_name:     string,
  parent_team_id:   Team.id | null,        // teams nest
  cost_center:      string,
}

// ── Edge type (one shape, four kinds) ────────────────────────────────

Edge {
  type: "reports_to" | "manages" | "member_of_team"
       | "compensated_at_band" | "sponsored_by" | "audits",
  source:           NodeId,
  target:           NodeId,
  valid_from:       date,
  valid_until:      date | null,            // null = currently active
  attributes:       object,                // edge-kind specific
}

// All writes are append-only. Edges are never deleted —
// they are closed by setting valid_until. That is what makes
// "who reported to whom in Q2 of last year" a one-line query.
§ 03Why a graph, not a table

The org has not been a tree for a long time.

A traditional HRIS represents the org as a tree. One employee, one manager, one location, one cost center. The tree was correct in 2008. By 2026 the org is a graph: matrix relationships, dotted lines, contractor pools, vendor liaisons, AI agents that own workflows, and reorgs every six months whether you wanted one or not.

We chose a graph because the questions our customers ask demand it. "Who is exposed to this risk?" is a graph traversal, not a table join. "What is our manager span of control by band?" aggregates over an edge type. "Who has access to this candidate's file, today, transitively?" is a many-hop walk with valid_from and valid_until constraints. Pretending those questions fit in a star schema is how every legacy HRIS slowly accreted a thousand custom reports — each of them a denormalized graph traversal hand-rolled by a Workday admin on a Tuesday afternoon.

The graph is also forward-compatible. As new entity types appear — more sophisticated AI agents that own goals, contractor-of- contractor relationships when a vendor sublets, project-based team memberships that last six weeks and then dissolve — we add node and edge types without breaking the existing schema. The same query that counted employees yesterday counts agents tomorrow, because both are nodes and the question was always "what kind of thing reports to what."

And because every edge carries valid_from and valid_until, the graph is bitemporal. We can answer who reported to whom on March 14th with the same machinery that answers who reports to whom today. The org is a graph; the graph is versioned; the version is a first-class citizen. That is the substrate the Console queries, the agents write to, and the Compact reads from when it ratifies a decision.
